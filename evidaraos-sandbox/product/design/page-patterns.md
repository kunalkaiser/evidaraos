# EvidaraOS Page Patterns

## Homepage

Use a first-viewport hero with a product architecture or evidence operations console. The visitor should understand that EvidaraOS is an evidence workflow platform, not a chatbot.

Required sections:

- hero
- problem
- platform modules
- how it works
- governance
- use cases
- demo CTA

Enterprise-health-intelligence upgrade:

- Add a proof mosaic using EvidaraOS artifacts, demo projects, module outputs, and research/methodology cards.
- Add a "The Work" band that explains how raw biomedical evidence becomes governed evidence artifacts.
- Add a "Platform modules" band that separates product modules from technical skills.
- Add a "Resources" band for methodology notes, validation guides, and demo artifacts.
- Avoid fake impact numbers until real validation or customer evidence exists.

## Platform Page

Show EvidaraOS as a governed evidence intelligence layer with modules connected to a common workflow lifecycle.

Use:

- system stack diagram
- workflow lifecycle
- module matrix
- validation and audit controls

## Solution Pages

Every solution page should show:

- product module name
- primary users
- workflow
- artifacts produced
- guardrails
- workspace status

Do not expose raw skill internals as marketing.

Recommended solution-page flow:

1. Hero with product module and primary job.
2. Evidence problem and why it matters.
3. Workflow architecture.
4. Artifacts produced.
5. Governance and validation controls.
6. What requires human review.
7. Fixture demo status or live execution status.
8. CTA.

## Workspace

Use dense operational layouts:

- project cards
- module index
- status chips
- artifact tables
- review queues
- validation reports
- audit trails

If live execution is pending, label it honestly.
